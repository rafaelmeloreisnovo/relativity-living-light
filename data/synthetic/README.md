# Synthetic data

Synthetic inputs belong here when they are used for code sanity, examples, numerical stability checks, or regression tests. They cannot support scientific claims.

## Required metadata

Serialized synthetic JSON artifacts should include:

- `dataset_type: synthetic_sanity_check` or `dataset_type: synthetic_regression_test`;
- `claim_allowed: false` or an equivalent boundary policy that blocks claims;
- enough provenance to reproduce how the file was generated.

## Approved roots

Synthetic/mock/demo/example/fixture artifacts must live under one of the approved boundary roots:

- `data/synthetic/`
- `results/synthetic/`
- `tests/fixtures/`

Structure-D generated regression contracts now use `data/synthetic/structure_d/mock_data_contract.json` as the canonical synthetic contract path. `data/inputs/structure_d/mock_data_contract.json` is retained only as a temporary compatibility alias for older tests/scripts.

## Binary artifact policy

Binary render artifacts such as PNG figures are not duplicated in source-only migrations. Their approved `results/synthetic/figures/` destinations remain recorded in `LEGACY_SYNTHETIC_MANIFEST.json`, but the files must be regenerated by CI/source workflows rather than committed as newly copied binaries.

## Legacy manifest

`LEGACY_SYNTHETIC_MANIFEST.json` is the migration ledger for legacy synthetic/mock/demo/example/fixture artifacts that still need temporary compatibility aliases outside the approved roots. Each entry records the original alias and the approved boundary path.

Every manifest entry has `scientific_use_allowed: false`. `code_test_use_allowed` only means the artifact can be used for tests or diagnostics; it does not grant scientific use. Remove a legacy exception only after confirming no real-data pipeline depends on the old path.

## CI gate

Run the synthetic path boundary gate before merging boundary changes:

```bash
python -m data.pipelines.structure_d.synthetic_real_boundary
```
